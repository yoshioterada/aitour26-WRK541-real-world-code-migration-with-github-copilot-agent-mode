# 参加者からのよくある質問 – プロクター向け FAQ

ワークショップ中によくある質問への回答テンプレートです。参加者へ一貫したサポートを提供するためのクイックリファレンスとして活用してください。

## GitHub Copilot に関する質問

### Q: 「Copilot が何も提案してくれません」

**クイックチェック:**

- VS Code のステータスバーに Copilot アイコンは表示されていますか？
- ファイルは保存されていますか？（保存済みの方が精度が上がります）
- 文脈は十分ですか？関連ファイルを開く、または `#file:` を使って参照を与えましょう

**トラブルシュート手順:**

1. Copilot がアクティブか確認（アイコンにエラーなし）
2. モデルを切り替えてみる
3. VS Code をリロード（`Ctrl+Shift+P` → "Reload Window"）
4. ネット接続を確認
5. GitHub サインアウト/サインイン

**回答例:**
「ステータスバーの Copilot アイコンは見えますか？ファイルを保存して、`#file:` 参照で文脈を増やしてみましょう。」

---

### Q: 「別のモデルは使えますか？どれが一番いい？」

**回答:**
「はい、複数モデルが使えます。このワークショップでは **GPT-4o-mini** で検証済みですが、
Claude や Gemini も試せます。**GPT-4o** は複雑なタスクに正確、**GPT-4o-mini** は高速です。
場面に応じて選んでください。」

**切り替え方法:**

- Copilot Chat のモデル名をクリック
- 利用可能なモデルを選択
- 提案が微妙なら別モデルを試す

---

### Q: 「Agent Mode はチートですか？自分で書くべき？」

**回答:**
「Agent Mode はチートではありません。熟練エンジニアとのペアプロを模したツールです。学びのポイントは:

- 何を依頼するか
- 生成コードの検証
- 動かないときのデバッグ
- 信頼すべきか検証すべきかの見極め

AI とペアプロしているだけで、判断はあなたが行います。」

---

### Q: 「Copilot の提案が手順と違う。どうすれば？」

**回答:**
「良い学習機会です！以下を試してください:

1. **検証**: 期待動作と比較
2. **プロンプト改善**: 具体的に指示
3. **反復**: フィードバックして修正を依頼
4. **検証**: テストを実行

AI 提案は完璧ではありません。受け入れ/修正/拒否の判断力が重要です。」

---

## テストに関する質問

### Q: 「Copilot の通りにやったのにテストが落ちます」

**トラブルシュート:**

1. **アプリは起動していますか？**
   - Python テストはアプリ起動が必須
   - `curl http://localhost:8000` が応答するか確認

2. **ポートは正しいですか？**
   - Python アプリ: 8000
   - C# アプリも 8000 で動かす（互換性のため）
   - `dotnet run --urls "http://localhost:8000"`

3. **BASE_URL は設定済み？**
   - Windows: `$env:BASE_URL="http://localhost:8000"`
   - Linux/Mac: `export BASE_URL="http://localhost:8000"`

4. **テスト出力を確認**
   - 失敗したアサーションを読む
   - 期待値 vs 実際の値を比較
   - エラーメッセージを Copilot に聞く

**回答例:**
「一緒に見ましょう。アプリは起動してますか？テスト出力を見せてください。」

---

### Q: 「他のテストフレームワーク（xUnit/NUnit）は使えますか？」

**回答:**
「使えます。ただし **Python テストは必須** です（互換性検証のため）。C# テスト（MSTest/xUnit/NUnit）は任意追加です。」

---

### Q: 「全エンドポイントのテストを書く必要がありますか？」

**回答:**
「Python テストが HTTP 経由で全エンドポイントを検証します。

- **必須**: Python テスト合格（移行の正しさ）
- **任意**: C# ユニットテスト追加（練習に最適）」

## 環境 & セットアップ

### Q: 「Codespaces とローカル、どちらが良い？」

**回答:**
「**Codespaces 推奨** 理由:

- ✅ セットアップ不要
- ✅ 環境が全員で共通
- ✅ ブラウザがあれば動作
- ✅ インストール不要

ローカルは Python 3.12 と .NET 10 SDK が整っていればOK。」

---

### Q: 「Codespace が重い/固まる」

**対処:**

1. ページ再読み込み
2. Codespace 停止 → 再起動
3. 新しい Codespace 作成（最終手段）

**予防:**

- 使わないタブを閉じる
- 不要なプロセスを停止
- サーバーは `Ctrl+C` で止める

**回答例:**
「まずページを再読み込みしましょう。ダメなら Codespace を再起動します。」

---

### Q: 「ポート 8000 が使用中」

**対処:**

Windows PowerShell:

```powershell
Get-NetTCPConnection -LocalPort 8000
Stop-Process -Id <PID> -Force
```

Linux/Mac:

```bash
lsof -i :8000
kill -9 <PID>
```

**別ポートを使う:**

```bash
# Python
uvicorn main:app --port 8001

# C#
dotnet run --urls "http://localhost:8001"
```

**回答例:**
「ポート使用プロセスを止めましょう。別ポートを使う場合は BASE_URL も忘れずに更新してください。」

## コード移行に関する質問

### Q: 「どの Copilot モードを使うべき？」

**クイックガイド:**

| タスク | モード | 理由 |
| ------ | ------ | ------ |
| コード理解・質問 | **Ask** | 学習のみ、変更なし |
| C# スキャフォールディング作成 | **Agent** | コマンド含む多段タスク |
| 移行計画を立てる | **Plan** | 実装前の戦略策定 |

**回答例:**
「質問は Ask、小規模変更でも多段タスクでも Agent、戦略は Plan が目安です。」

---

### Q: 「C# 以外へ移行してもいい？」

**回答:**
「ワークショップは Python→C# 用ですが、学ぶパターン（段階的移行・テスト検証・AI活用）は他言語にも応用できます。ぜひ Java/Go/Rust などにもチャレンジを。」

---

### Q: 「なぜ Python テストを保持するの？C# テストに書き換えない？」

**回答:**
「Python テストが”正解”だからです:

1. 元の動作を検証する
2. HTTP 挙動（ユーザー視点）を確認する
3. 新APIが互換であることを証明する
4. 言語非依存でどの実装にも使える

実際のリライトでも元のテストで検証する戦略は有効です。」

---

## GitHub Copilot のファイル参照

### Q: 「プロンプトの `#file:` って何？」

**回答:**
「`#file:` は Copilot に対象ファイルを明示する記法です。

1. Copilot チャットで `#` を入力
2. ファイルピッカーでファイルを選択
3. 残りのプロンプトを入力

ファイル文脈を与えることで精度が上がります。」

**デモ:** `#` → ファイル選択 → プロンプト入力

---

### Q: 「毎回 `#file:` を使う必要ありますか？」

**回答:**
「必須ではありません。以下のときに使うと良いです:

- 開いていない特定ファイルに対して作業するとき
- Copilot にファイル文脈が必要なとき
- 精度を高めたいとき

一般的な質問や選択中のファイルには不要です。」

## 時間管理に関する質問

### Q: 「遅れています。何を優先して進めるべき？」

**優先順位:**

1. **必須**
   - Python プロジェクト理解
   - C# スキャフォールディング
   - 2〜3 エンドポイント移行
   - Python テストを通す

2. **できれば**
   - 全エンドポイント移行
   - 全テスト通過
   - 検証プロセス理解

3. **時間があれば**
   - C# テスト追加
   - コード最適化
   - チャレンジ 1/2

**回答例:**
「スキャフォールディングと 2〜3 エンドポイントのテスト合格を最優先にしましょう。残りは後で取り組めます。」

---

### Q: 「早く終わりました。次は？」

**提案:**

1. チャレンジ 1: コンテナ化（🔥 中級）
2. チャレンジ 2: DB 統合（🔥🔥 上級）
3. C# ユニットテスト追加（MSTest）
4. コード最適化（Copilot に改善提案を聞く）
5. 他の参加者を手伝う（教えることで定着）
6. 別モデルを試して挙動を比較

**回答例:**
「素晴らしい！チャレンジ 1/2 や C# テスト追加を試してみてください。別モデル比較も面白いですよ。」

## 一般トラブルシュート

### Q: 「何も上手くいきません。やり直せますか？」

**やり直す前に:**

1. `git status` / `git diff` で最近の変更を確認
2. `git checkout -- <file>` で直前の変更を戻す
3. パス/ポートなど簡単な環境問題か確認
4. プロクターと 5 分ペアデバッグ

**やり直す場合:**

- 新しい Codespace を作る
- または `git reset --hard` で最後のコミットへ戻す
- `src/csharp-app-complete` の完成例を参照

**回答例:**
「まず修正を試してみましょう。完全にやり直すと学びが減るので、5分だけ一緒にデバッグさせてください。」

## プロクター向けヒント

### スタックした参加者への対応

1. 問題を口頭で説明してもらう（説明中に気づくことも多い）
2. 基本を確認（アプリ起動？ディレクトリ？ファイル保存？）
3. エラーメッセージを一緒に読む（デバッグ力を育てる）
4. Copilot にエラーを尋ねる方法を示す
5. 修正ではなく「伴走」する

### 同じ問題が複数発生

- 全体でアナウンス
- 「Common Issues」ボードに追加
- トラブルシュートドキュメントを更新
- 手順書の改善を検討

### 良い習慣の喚起

- ✅ 変更ごとに検証
- ✅ エラーをよく読む
- ✅ Copilot でエラー解説を依頼
- ✅ 小さくテスト
- ✅ 早めに質問
